Spesifikasi Kebutuhan Perangkat Lunak (SKPL)

Dokumen ini merinci 18 Use Case dengan alur lengkap (Pre-condition, Flow, Post-condition) sesuai standar SKPL.

#1 Membuat atau Masuk Akun
Aktor: Guest, User

Deskripsi

Mencocokkan data user dengan database; jika belum ada, user menginputkan biodata baru.

Pre-condition

  • Pengunjung belum login (Guest).

Primary Flow

  1. User mengakses halaman Login.
  2. User memasukkan username dan password.
  3. Sistem memvalidasi kredensial.
  4. Jika valid, sistem membuat sesi dan redirect ke Home.
  5. Jika belum punya akun, user memilih "Register".
  6. User mengisi form registrasi (Nama, Username, Password).
  7. Sistem menyimpan data user baru.

Post-condition

  • User berhasil login dan mendapatkan akses sesuai Role.
#2 Mereset Password
Aktor: User

Deskripsi

User membuat password baru jika lupa password lama.

Primary Flow

  1. User memilih "Lupa Password" di halaman login.
  2. User memasukkan Token Reset yang valid (didapat dari Admin).
  3. User memasukkan password baru dan konfirmasi.
  4. Sistem mengupdate password di database.
#3 Mengelola Profil
Aktor: User, Admin

Deskripsi

User mengubah data profil seperti Nama, Username, atau Password.

Pre-condition

  • User telah login.

Primary Flow

  1. User masuk ke halaman Profil.
  2. User mengedit kolom yang diizinkan (Nama, Foto).
  3. User menekan tombol Simpan.
  4. Sistem memvalidasi input (unik username, dll).
  5. Sistem menyimpan perubahan.
#4 Mencari Fasilitas & #18 Menyaring Pencarian
Aktor: User, Guest

Deskripsi

Mencari fasilitas berdasarkan nama dan menyaring berdasarkan Tipe (Rapat, Aula, Lab).

Primary Flow

  1. User mengetik kata kunci di Search Bar.
  2. User memilih filter Tipe Fasilitas (opsional).
  3. Sistem melakukan query pencarian (ILIKE) dengan filter.
  4. Sistem menampilkan daftar fasilitas yang cocok.
#7 Mengajukan Reservasi
Aktor: User (Mahasiswa/Dosen)

Deskripsi

Mengajukan peminjaman ruangan dengan detail kegiatan.

Pre-condition

  • User login. Fasilitas tersedia.

Primary Flow

  1. User memilih tanggal dan jam pada kalender fasilitas.
  2. Sistem memvalidasi ketersediaan slot (Check Conflict).
  3. User mengisi Form: Keperluan, Jumlah Peserta.
  4. User mengunggah Dokumen Proposal (Opsional).
  5. User submit reservasi.
  6. Sistem menyimpan data dengan status PENDING.
#9 Menyetujui atau Menolak Reservasi
Aktor: Admin

Deskripsi

Admin memverifikasi pengajuan reservasi.

Pre-condition

  • Ada reservasi baru berstatus PENDING.

Primary Flow

  1. Admin melihat daftar reservasi Pending.
  2. Admin memeriksa detail dan dokumen proposal.
  3. Admin memilih: Approve atau Reject.
  4. Sistem mengupdate status reservasi.
  5. Sistem mengirim notifikasi ke User yang bersangkutan.
#13 Mengelola Fasilitas
Aktor: Admin

Deskripsi

CRUD (Create, Read, Update, Delete) data fasilitas gedung dan ruangan.

Primary Flow

  1. Admin masuk halaman Manage Facilities.
  2. Admin klik "Add Facility" untuk menambah.
  3. Admin mengisi Form: Nama, Kapasitas, Deskripsi, Foto.
  4. Sistem menyimpan data fasilitas baru.
