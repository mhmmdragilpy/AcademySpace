3.2.1.1 Use Case Scenario #13 <Menyetujui atau Menolak Reservasi> ADMIN

Skenario Use Case #13:

i. Pre-Condition
  • Admin sudah login dengan role Admin.
  • Terdapat minimal satu reservasi dengan status "Pending".
ii. Use Case Description
a. Primary Flow (Melihat Daftar Reservasi)
  • Admin mengakses menu "Reservations" dari sidebar.
  • Halaman Manage Reservations ditampilkan.
  • Admin dapat memfilter reservasi berdasarkan status (All, Pending, Approved, Rejected).
  • Tabel ditampilkan dengan informasi:
    • Nama User dan Username
    • Fasilitas yang direservasi
    • Tanggal dan Waktu
    • Status
    • Dokumen pendukung (jika ada)
    • Tombol aksi
b. Primary Flow (Menyetujui Reservasi)
  • Admin menemukan reservasi dengan status "Pending".
  • Admin mengklik tombol "View" untuk melihat dokumen pendukung (jika ada).
  • Admin mengklik tombol "Approve".
  • Dialog konfirmasi ditampilkan.
  • Admin mengkonfirmasi persetujuan.
  • Status reservasi diubah menjadi "Approved".
  • Notifikasi dikirim ke user bahwa reservasi disetujui.
c. Primary Flow (Menolak Reservasi)
  • Admin menemukan reservasi dengan status "Pending".
  • Admin mengklik tombol "Reject".
  • Dialog konfirmasi dengan field alasan penolakan (opsional) ditampilkan.
  • Admin mengisi alasan dan mengkonfirmasi penolakan.
  • Status reservasi diubah menjadi "Rejected".
  • Notifikasi dikirim ke user dengan alasan penolakan.
d. Alternative Flow
  • Tidak ada reservasi dengan status "Pending".
  • Tabel kosong atau pesan "Tidak ada reservasi pending" ditampilkan.
iii. Post-Condition
  • Status reservasi berubah menjadi "Approved" atau "Rejected".
  • User menerima notifikasi tentang keputusan admin.
  • Jika approved, slot waktu terkunci untuk reservasi tersebut.
  • Jika rejected, slot waktu kembali available.
