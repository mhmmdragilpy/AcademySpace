3.1.1.8 Use Case Scenario #8 <Mengedit atau Membatalkan Reservasi>

Skenario Use Case #8:

i. Pre-Condition
  • User sudah login.
  • User memiliki minimal satu reservasi aktif (status Pending atau Approved).
  • Reservasi belum melewati tanggal pelaksanaan.
ii. Use Case Description
a. Primary Flow (Mengedit Reservasi)
  • User mengakses halaman "My Reservations".
  • User mencari reservasi yang ingin diedit.
  • User mengklik tombol "Edit" pada reservasi tersebut.
  • Form edit dengan data reservasi saat ini ditampilkan.
  • User dapat mengubah:
    • Fasilitas (dropdown)
    • Tanggal reservasi
    • Waktu mulai dan selesai
    • Catatan/keperluan
  • User mengklik tombol "Save Changes".
  • Perubahan divalidasi.
  • Perubahan disimpan dan status menjadi "Pending" (jika sebelumnya Approved).
  • User diarahkan kembali ke halaman My Reservations.
b. Primary Flow (Membatalkan Reservasi)
  • User mengakses halaman "My Reservations".
  • User mencari reservasi yang ingin dibatalkan.
  • User mengklik tombol "Cancel" pada reservasi tersebut.
  • Dialog konfirmasi ditampilkan.
  • User mengkonfirmasi pembatalan.
  • Status reservasi diubah menjadi "Cancelled".
  • Notifikasi sukses ditampilkan.
c. Alternative Flow
  • User mencoba mengedit reservasi yang sudah lewat tanggalnya.
  • Pesan bahwa reservasi tidak dapat diedit ditampilkan.
  • User mencoba mengedit ke slot waktu yang sudah terisi.
  • Pesan error "Slot waktu tidak tersedia" ditampilkan.
iii. Post-Condition
  • Reservasi berhasil diubah atau dibatalkan.
  • Slot waktu yang dibatalkan kembali tersedia untuk booking lain.
  • Admin menerima notifikasi jika reservasi yang sudah di-approve di-edit.
